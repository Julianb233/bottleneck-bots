## Goal

Create `server/services/ghl.service.ts` with OAuth 2.0 token management, auto-refresh, rate limiting, and a typed API client wrapper. Build on the existing `credentialVault.service.ts` for encrypted token storage and the existing `server/_core/env.ts` for configuration.

### 2. Create GHL Service (`server/services/ghl.service.ts`)

Core class `GHLService` with:

**OAuth Manager**:
- `initiateAuthorization(userId: number, scopes: string[])` -- generate authorization URL with PKCE (code_verifier + code_challenge), store state in `oauthState.service.ts`, return auth URL
- `handleCallback(code: string, state: string)` -- exchange code for tokens, verify state, store encrypted tokens via `credentialVault.service.ts` with service=`ghl`, type=`oauth_token`
- `revokeAccess(userId: number, locationId: string)` -- revoke token at GHL API, delete from credential vault

**Token Manager**:
- `getAccessToken(userId: number, locationId: string)` -- retrieve from vault, check expiry, auto-refresh if within 5 minutes of expiration
- `refreshToken(userId: number, locationId: string)` -- POST to `https://services.leadconnectorhq.com/oauth/token` with grant_type=refresh_token, update vault

**API Client**:
- `request<T>(config: { method, endpoint, locationId, userId, data?, params? })` -- authenticated HTTP call with:
  - Auto-attach Bearer token (via `getAccessToken`)
  - Rate limiting (token bucket per locationId, respect GHL's 100 req/min per location)
  - Retry with exponential backoff on 429/5xx (max 3 retries)
  - Circuit breaker (open after 5 consecutive failures, half-open after 30s)
  - Response typing with `GHLApiResponse<T>` wrapper including rate limit headers
- `batchRequest<T>(requests[])` -- serial execution with rate limit awareness

**Rate Limiter** (in-memory, upgradeable to Redis):
- Token bucket algorithm: 100 tokens/minute per location
- Track via `Map<locationId, { tokens, lastRefill }>`
- Queue requests when bucket empty, resume on refill

### 3. Create GHL Drizzle Schema

**File**: `drizzle/schema-ghl.ts`

Tables:
- `ghlConnections` -- userId, locationId, companyId, locationName, status (connected/disconnected/error), scopes, connectedAt, lastSyncAt
- `ghlWebhookEvents` -- id, locationId, eventType, eventId (for dedup), payload (jsonb), status (pending/processed/failed/dead_letter), processedAt, error, attempts, createdAt
- `ghlSyncLog` -- id, userId, locationId, operation, entityType, entityId, direction (inbound/outbound), status, error, createdAt

Update `drizzle.config.ts` to include `schema-ghl.ts`.

### 4. Create GHL tRPC Router

**File**: `server/api/routers/ghl.ts`

Procedures (all `protectedProcedure`):
- `ghl.connect` -- input: `{ scopes?: string[] }` -- calls `initiateAuthorization`, returns `{ authorizationUrl }`
- `ghl.callback` -- input: `{ code, state }` -- calls `handleCallback`, returns connection status
- `ghl.disconnect` -- input: `{ locationId }` -- calls `revokeAccess`
- `ghl.status` -- no input -- returns list of connected locations with health status
- `ghl.listLocations` -- no input -- returns all connected locations for user
- `ghl.testConnection` -- input: `{ locationId }` -- makes a test API call to verify tokens work

Register in `server/routers.ts` alongside existing routers like `scheduledTasks`, `templates`, etc.

### 5. Create OAuth Express Routes

**File**: `server/api/routes/ghl-oauth.ts`

These must be Express routes (not tRPC) because GHL redirects back to a URL:
- `GET /api/ghl/oauth/callback` -- handles OAuth redirect from GHL, exchanges code, redirects user back to Settings page with success/error query param

Register in `server/_core/index.ts` where other Express routes are mounted.

### 6. Add GHL Connection UI to Settings

**File**: `client/src/pages/Settings.tsx` (enhance existing)

Add a "GoHighLevel" section/tab:
- "Connect GoHighLevel" button (calls `ghl.connect`, redirects to GHL auth)
- Connected locations list with status badges
- Disconnect button per location
- "Test Connection" button
- Last sync timestamp

2. After authorizing -> redirected back to Settings page with success message

4. `ghl.testConnection` makes a GET to `/contacts/v1/` and returns success
5. Token auto-refreshes when within 5 min of expiry (verify with short-lived test token)
6. Rate limiter queues requests when > 100/min for a single location
7. Disconnecting revokes the token and removes from vault

## Risks

- GHL OAuth redirect requires a publicly accessible callback URL (Vercel handles this)
- Rate limits are per-location; must track accurately to avoid 429s
- Token refresh failures need graceful handling (mark connection as "needs re-auth")

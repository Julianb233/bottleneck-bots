### 2. Create GHL Webhook Processing Worker

**File**: `server/workers/ghl-webhook.worker.ts`

Event processing pipeline:
1. Read event from `ghlWebhookEvents` table
2. Route by event type (FR-023):

```typescript
const handlers: Record<GHLEventType, WebhookHandler> = {
  'contact.created':         handleContactCreated,
  'contact.updated':         handleContactUpdated,
  'contact.deleted':         handleContactDeleted,
  'contact.tag.added':       handleContactTagChanged,
  'contact.tag.removed':     handleContactTagChanged,
  'opportunity.created':     handleOpportunityCreated,
  'opportunity.updated':     handleOpportunityUpdated,
  'opportunity.stage.changed': handleOpportunityStageChanged,
  'appointment.created':     handleAppointmentCreated,
  'appointment.updated':     handleAppointmentUpdated,
  'appointment.cancelled':   handleAppointmentCancelled,
  'form.submitted':          handleFormSubmitted,
  'call.completed':          handleCallCompleted,
  'sms.received':            handleSmsReceived,
  'email.received':          handleEmailReceived,
};
```

3. Execute handler -> update event status to 'processed'
4. On failure: increment attempts, apply exponential backoff (FR-024)
   - Retry schedule: 10s, 30s, 2min, 10min, 1hr
   - After 5 failures: move to dead letter (status='dead_letter') (FR-025)
5. Log in `ghlSyncLog`

### 3. Event Handlers

Each handler follows the pattern:

**Contact events** -> sync contact data to local DB, trigger any matching user workflows
**Opportunity events** -> update local opportunity state, notify assigned users, trigger workflows
**Appointment events** -> sync to internal calendar, trigger reminder workflows
**Form submission** -> create/update contact, trigger lead capture workflows
**Message events** -> route to conversation inbox, trigger response workflows

Key handler: `handleFormSubmitted`:
- Extract form fields from payload
- Create or update contact (dedup by email/phone)
- Add source tag (form name)
- Trigger any workflows that have "Form Submission" as trigger node

Key handler: `handleOpportunityStageChanged`:
- Update opportunity in local DB
- Trigger workflows with "Pipeline Stage Changed" trigger
- Send notifications to assigned team member

### 8. Multi-Tenant Isolation (FR-026 through FR-030)

Enforce in every GHL service method:
- **Location Isolation (FR-026)**: Every API call includes locationId; service verifies user owns the location via `ghlConnections` table before executing
- **Agency Isolation (FR-027)**: User can only access their own GHL connections; enforced at tRPC `protectedProcedure` level
- **Credential Isolation (FR-028)**: Tokens stored per-user per-location in credential vault; vault enforces userId check
- **Audit Logging (FR-029)**: All GHL operations logged in `ghlSyncLog` with userId, locationId, operation
- **Access Control (FR-030)**: Check `agentPermissions.service.ts` before executing GHL operations (ghl_read, ghl_write permissions)

## Risks

- GHL webhook payloads vary by event type; need flexible parsing
- GHL webhook delivery can have delays during high-traffic periods
- Calendar API endpoints may differ between GHL v1 and v2
- BullMQ worker must handle restarts gracefully (don't lose queued events)
- Dead letter events may accumulate if GHL API changes structure

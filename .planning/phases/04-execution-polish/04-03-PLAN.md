## Tasks

### 1. Enhance Agent Orchestrator Error Handling

**File**: `server/services/agentOrchestrator.service.ts` (enhance the main agent loop)

Currently the orchestrator imports failure recovery and strategy adaptation but may not fully integrate them. Ensure:

**Error Classification** (using `errorTypes.ts`):
- On any tool execution failure, classify the error:
  - `element_not_found` -> retry with alternative selector (via selfCorrection)
  - `timeout` -> retry with increased timeout
  - `rate_limited` -> wait and retry (exponential backoff)
  - `element_not_interactable` -> scroll/wait, retry
  - `navigation_failed` -> retry navigation, try alternative URL
  - `authentication_required` -> pause execution, request re-auth
  - `captcha_detected` -> pause execution, alert user
  - `network_error` -> retry with backoff
  - `page_crashed` -> restart browser session, resume from checkpoint
  - `session_expired` -> create new session, resume from last checkpoint

**Automatic Retry Loop**:
```typescript
const MAX_RETRIES = 3;
const BACKOFF_BASE = 1000; // 1s, 2s, 4s

for (let attempt = 0; attempt <= MAX_RETRIES; attempt++) {
  try {
    result = await executeToolAction(action);
    break; // Success
  } catch (error) {
    const classified = classifyError(error);

    if (!isRecoverable(classified.type)) {
      // Fatal error - stop execution
      emitExplainedError(executionId, classified);
      throw error;
    }

    if (attempt === MAX_RETRIES) {
      // Try strategy adaptation before giving up
      const alternative = await strategyAdaptation.getAlternative(action, classified);
      if (alternative) {
        result = await executeToolAction(alternative);
      } else {
        emitExplainedError(executionId, classified);
        throw error;
      }
    }

    // Wait with backoff
    await delay(BACKOFF_BASE * Math.pow(2, attempt));

    // Apply recovery strategy
    const recovery = await failureRecovery.getRecoveryStrategy(classified);
    action = recovery.modifiedAction || action;
  }
}
```

**Screenshot on Error**:
- On any error during browser automation, capture a screenshot before retry
- Store screenshot URL on the execution step record
- Makes debugging much easier

### 2. Integrate Strategy Adaptation

**File**: `server/services/intelligence/strategyAdaptation.service.ts` (enhance)

When the standard retry loop exhausts retries:
1. Ask strategy adaptation for an alternative approach
2. Examples:
   - Browser click failed -> try keyboard navigation
   - Direct URL navigation failed -> try searching within the app
   - Form fill failed -> try tab-key navigation between fields
   - API call failed -> fall back to browser automation
3. If alternative succeeds, record it as a learning (for future pattern reuse via `memory/` services)

5. Click "Retry" on failed execution -> new execution starts with same task

7. Circuit breaker closes -> banner disappears
8. Screenshot captured on error -> visible in execution error display
9. 3+ same errors in 1 hour -> alert notification created

## Risks

- Aggressive retry can amplify problems (e.g., retrying a delete operation)
- Strategy adaptation requires LLM call, adding latency to error recovery
- Checkpoint resume may not work if page state changed between original execution and retry
- Circuit breaker state is per-process; if running multiple Vercel instances, state won't be shared (use Redis in production)

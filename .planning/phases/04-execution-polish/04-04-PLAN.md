## Tasks

### 1. Create Visual Cron Expression Builder Component

**File**: `client/src/components/scheduled/CronExpressionBuilder.tsx` (new)

A human-friendly cron builder that generates cron expressions:

**Mode: Simple** (default):
- Dropdown: "Every" -> Day / Weekday / Week / Month / Hour / Custom
- **Daily**: Time picker (hour + minute), timezone selector
- **Weekday**: Time picker, only Mon-Fri
- **Weekly**: Day-of-week checkboxes + time picker
- **Monthly**: Day-of-month dropdown (1-28) + time picker
- **Hourly**: Minute selector (every X minutes, or at specific minute)

**Mode: Advanced** (toggle):
- Raw cron expression input with live validation
- Syntax helper tooltip showing format: `minute hour day-of-month month day-of-week`
- Uses `cronSchedulerService.validateCronExpression` via tRPC

**Preview panel** (both modes):
- Human-readable description: "Runs every Monday at 9:00 AM PST"
- Next 5 execution times (calls `cronSchedulerService.getNextRunTimes`)
- Visual calendar mini-view showing upcoming runs this month

### 3. Enhance Scheduler Runner with BullMQ

**File**: `server/services/schedulerRunner.service.ts` (enhance)

Current state: picks up due tasks and executes. Enhance:

**Job Queue Integration**:
- Use BullMQ `Queue` for job scheduling instead of polling
- `scheduleTask(taskId, cronExpression)` -> creates repeatable BullMQ job
- `cancelSchedule(taskId)` -> removes repeatable job
- `runNow(taskId)` -> creates immediate one-off job

**Worker Processing**:
- BullMQ Worker processes jobs from `scheduled-agent-tasks` queue
- Each job: loads task config, starts agent execution, tracks result
- Handles missed executions: if server was down when task was due, detect and run on startup
- Concurrency control: max 3 concurrent scheduled executions per user (prevent resource abuse)

**Execution Recording**:
- Create `scheduledTaskExecutions` record before execution starts
- Update with status, duration, output, errors on completion
- Track `stepsCompleted` / `stepsTotal` for progress visibility

**Retry on Failure**:
- If `retryOnFailure` is enabled on the task:
  - Retry up to `maxRetries` times with `retryDelay` between attempts
  - Each retry is a new attempt number in the execution record
  - Stop retrying if any attempt succeeds
- If all retries fail: set task status to `failed`, send failure notification

## Risks

- BullMQ requires Redis; Vercel serverless doesn't have persistent Redis by default (need external Redis like Upstash)
- Missed execution detection requires comparing last run time with expected schedule
- Concurrent scheduled tasks could exhaust BrowserBase session limits
- Timezone handling is error-prone; always store cron in UTC and convert for display
- Long-running scheduled tasks could overlap with the next scheduled run
